29 // The following class defines a "semaphore" whose value is a non-negative
30 // integer. The semaphore has only two operations P() and V():
31 //
32 // P() -- waits until value > 0, then decrement
33 //
34 // V() -- increment, waking up a thread waiting in P() if necessary
35 //
36 // Note that the interface does *not* allow a thread to read the value of
37 // the semaphore directly -- even if you did read the value, the
38 // only thing you would know is what the value used to be. You don't
39 // know what the value is now, because by the time you get the value
40 // into a register, a context switch might have occurred,
41 // and some other thread might have called P or V, so the true value might
42 // now be different.
43

93 // The following class defines a "condition variable". A condition
94 // variable does not have a value, but threads may be queued, waiting
95 // on the variable. These are only operations on a condition variable:
96 //
97 // Wait() -- release the lock, relinquish the CPU until signaled,
98 // then re-acquire the lock
99 //
100 // Signal() -- wake up a thread, if there are any waiting on
101 // the condition
102 //
103 // Broadcast() -- wake up all threads waiting on the condition
104 //
105 // All operations on a condition variable must be made while
106 // the current thread has acquired a lock. Indeed, all accesses
107 // to a given condition variable must be protected by the same lock.
108 // In other words, mutual exclusion must be enforced among threads calling
109 // the condition variable operations.
110 //
111 // In Nachos, condition variables are assumed to obey *Mesa*-style
112 // semantics. When a Signal or Broadcast wakes up another thread,
113 // it simply puts the thread on the ready list, and it is the responsibility
114 // of the woken thread to re-acquire the lock (this re-acquire is
115 // taken care of within Wait()). By contrast, some define condition
116 // variables according to *Hoare*-style semantics -- where the signalling
117 // thread gives up control over the lock and the CPU to the woken thread,
118 // which runs immediately and gives back control over the lock to the
119 // signaller when the woken thread leaves the critical section.
120 //
121 // The consequence of using Mesa-style semantics is that some other thread
122 // can acquire the lock, and change data structures, before the woken
123 // thread gets a chance to run.
124
