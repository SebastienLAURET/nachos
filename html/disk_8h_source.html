2 // Data structures to emulate a physical disk. A physical disk
3 // can accept (one at a time) requests to read/write a disk sector;
4 // when the request is satisfied, the CPU gets an interrupt, and

25 // The following class defines a physical disk I/O device. The disk
26 // has a single surface, split up into "tracks", and each track split
27 // up into "sectors" (the same number of sectors on each track, and each
28 // sector has the same number of bytes of storage).
29 //
30 // Addressing is by sector number -- each sector on the disk is given
31 // a unique number: track * SectorsPerTrack + offset within a track.
32 //
33 // As with other I/O devices, the raw physical disk is an asynchronous device --
34 // requests to read or write portions of the disk return immediately,
35 // and an interrupt is invoked later to signal that the operation completed.
36 //
37 // The physical disk is in fact simulated via operations on a UNIX file.
38 //
39 // To make life a little more realistic, the simulated time for
40 // each operation reflects a "track buffer" -- RAM to store the contents
41 // of the current track as the disk head passes by. The idea is that the
42 // disk always transfers to the track buffer, in case that data is requested
43 // later on. This has the benefit of eliminating the need for
44 // "skip-sector" scheduling -- a read request which comes in shortly after
45 // the head has passed the beginning of the sector can be satisfied more
46 // quickly, because its contents are in the track buffer. Most
47 // disks these days now come with a track buffer.
48 //
49 // The track buffer simulation can be disabled by compiling with -DNOTRACKBUF
50
